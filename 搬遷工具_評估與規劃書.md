# i18n-first 舊站搬遷工具 — 評估與規劃書

> 承接《網站改版討論》分享對話的構想:把香雲寺的爬蟲/搬遷工作,做成一個**開源的 .NET 多語言舊站搬遷工具**。
> 本文由「已實際完成香雲寺整套搬遷」的角度撰寫 —— 評估與規劃都用真實踩過的坑背書,不是紙上談兵。

> 2026-07-18 補充:**新增 Part 3 工程審查**(編碼、Phase 契約、語言代碼、跨平台、測試策略等),並據此修訂路線圖(§3.11)。

# Part 1 — 評估

## 1.1 一句話結論

**值得做,但不是「.NET 版 Firecrawl」,而是「唯一認真處理多語言遷移的工具,剛好用 .NET 寫」。** 市場空位是真的,而且我們手上就有一個完整、可展示的 case(香雲寺)當作 1.0 的背書 —— 這是任何競品都沒有的起手優勢。

## 1.2 市場空位(對話已查證,我補充驗證)

| 層級 | 現況 | 我方策略 |
|---|---|---|
| 單頁 HTML→MD 轉換器 | 紅海(Turndown/Pandoc/markdownify…) | 當函式庫用掉,不競爭 |
| 通用爬蟲/鏡像 | 成熟(wget/HTTrack/Scrapy/Screaming Frog) | 當現成元件,**不重造** |
| 整站遷移 API | **Firecrawl**(YC、14 萬星、LLM 抽欄位) | 主要對手,但有縫 |
| 多語言遷移管線 | **沒有任何產品做** | ← **這就是空位** |

**Firecrawl 的縫(= 我方的楔子):**
1. 付費雲端 API、要金鑰 —— 我方**自架、離線、可重跑**。
2. 通用、不吃 per-site config —— 我方**一站一份設定檔**。
3. **不做雙語配對** —— 這是我方獨門。
4. 全 Python/JS —— **.NET 完全空白**。

## 1.3 ⚠️ 最大風險:受眾錯位(對話已點出,我同意且加碼)

.NET 這塊沒競爭者,一部分是因為**會做「PHP 舊站→靜態站」遷移的人,大多本來就在 JS/Python 生態**,要工具直覺是 `npm`/`pip`,不會想到 NuGet。所以「無競爭」是雙面刃 —— green field,但自然客群可能不在這片地上。

**結論:不能靠「.NET 寫的」當賣點**(對使用者不是價值)。唯一能拉人的是那個獨門功能:**i18n-first**。任何多語言機構站(政府、大學、NGO、宗教組織)遷移時都在**手工對配多語版本**,非常痛,而且沒有工具解。

## 1.4 我用香雲寺實作驗證了什麼(這段是我方的獨有籌碼)

我把整套跑完了,證實兩件對產品定位關鍵的事:

**✅ 驗證了核心價值(雙語配對真的省事):**
香雲寺 `/ch/{path}` 與 `/en/{path}` 對稱,新聞用**去語言前綴的路徑當 `translation_key`** 就自動配對,220 篇中英一秒配好,完全不用模糊比對。**這確實是市面沒有、且真的痛的功能。**

**⚠️ 但也發現了對話沒料到的複雜度(這反而是產品的護城河):**
> **「對稱 URL 自動配對」只對「檔名對稱」的內容類型成立。** 香雲寺**新聞**檔名是日期(中英同名)→ 完美自動配;但**活動**檔名是語意命名且中英不同(中 `2026_cjgx` / 英 `enChant`)→ **自動配不起來,需要人工或啟發式配對**。

這個發現很重要:**天真的「URL 對調」會在真實站上失敗一半**。一個好的工具必須:
1. 自動配對「能配的」(對稱檔名)。
2. **偵測並列出「配不起來的」**,提供人工覆核介面(產出 `content_inventory.csv` 的 has_ch/has_en 缺漏表)。
3. 支援用「內容相似度 / 共用媒體 / 日期」做啟發式建議配對。

**這正是能拉開與 Firecrawl 差距的地方 —— 它連單語都用 LLM 硬抽,更不會處理這種雙語對配的邊角。**

## 1.5 可行性:已被香雲寺實作證明

整條管線我都跑過、能運作(495 頁、4.6GB 媒體、231 篇雙語新聞、全站巡檢 0 錯誤)。**技術風險基本清零** —— 剩下的是把 Python 原型的邏輯移植成 .NET + 抽象成 config,不是研究未知。

# Part 2 — 規劃書

## 2.1 定位(一句話)

> **The i18n-first static-site migrator** — 一個 config 驅動、可離線重跑的舊站→靜態站搬遷工具,天生為多語言而生,用 .NET 實作。

## 2.2 產品形態(兩層,一次餵滿兩種使用者)

1. **`dotnet tool` 全域 CLI**(給「只想用」的人)
   `dotnet tool install -g polymigrate` → 吃一份 site config → 吐出標準化 Markdown + 三份清單。
2. **NuGet library**(給要嵌進自己 pipeline 的人 + 你的 portfolio)
   把抽取/配對/盤點邏輯開放為可組合元件。

## 2.3 核心決策:雙語做成「核心」不是「外掛」

**建議做成核心功能**(工具天生假設多語言,單語言是特例)。理由:
- 定位最鋒利、最好行銷(「the i18n-first migrator」),這是唯一記憶點。
- 我方 case 就是雙語,不做核心等於丟掉最強籌碼。
- 代價:單語使用者覺得被綁 —— 但用「單語 = 只有一個 locale 的特例」在架構上化解即可(核心仍能跑單語,只是不觸發配對)。

## 2.4 管線設計(直接來自香雲寺實跑的四階段)

```
Phase 0  環境診斷    → 診斷 bot 防護/渲染方式/sitemap/正文 selector,寫進 config
Phase 1  鏡像+盤點   → (用現成爬蟲)全站抓下、URL 清單 = source of truth
Phase 2  結構化抽取  → HTML → 帶 frontmatter 的 Markdown + 三份清單 + 雙語配對
Phase 3  匯出        → 中性格式,可餵 Astro/Hugo/Umbraco 等
```

**工具的價值集中在 Phase 0 診斷輔助 + Phase 2 抽取管線**(Phase 1 用 Abot/wget 當元件)。

## 2.5 一站一份 Config(該抽象的東西 —— 全部來自香雲寺真實需求)

```yaml
site:
  base_url: https://www.ibps-austin.org
  # bot 防護繞法(香雲寺:JS cookie 挑戰 → 帶 cookie)
  auth_workaround: { type: cookie, set: { humans_21909: "1" } }
  render: static            # static | headless(需 JS 才長出正文時)
  polite: { concurrency: 1, delay_ms: 3000 }

url_pattern:
  # /{lang}/{section}/{page}.php
  lang_prefix: [ch, en]
  default_lang: ch

extract:
  # 正文 = 非框架的 <section>(香雲寺實測規則)
  content: "section[id]:not(#header):not(#footer):not(#header_main):not(#footer_main)"
  title: "#personal h3, .personal_1 h3"   # 優先內文 h3,非 <title>(常有雜訊)
  strip_title_prefix: ['\\d{4}/\\d{2}/\\d{2}\\s*新聞[:：]', 'News.*[:：]']

pairing:
  strategy: symmetric_path   # 去語言前綴的路徑當 translation_key
  # 配不起來的(如活動)→ 啟發式建議 + 人工覆核清單
  fallback: [shared_media, date, title_similarity]

media:
  download: true
  thumbnails: { max_width: 1000, quality: 82 }
```

## 2.6 ★ 把香雲寺踩過的坑「內建成功能」(這是別人抄不到的護城河)

這些是我實際遇到、修過的問題,每一個做成工具的內建處理,就是與通用工具的差距:

| 踩過的坑 | 內建處理 |
|---|---|
| bot 防護回 409(JS cookie 挑戰,非 UA) | config 宣告繞法;Phase 0 自動診斷 |
| 標題含冒號 → YAML 壞 | frontmatter 一律用 YAML lib 正確跳脫 |
| 數字 slug 前導 0 遺失(PyYAML vs js-yaml 判斷不一致) | 數字型字串強制加引號 |
| 圖片路徑 `%20` 雙重編碼 | 磁碟存解碼名、URL 單次編碼 |
| **手機直拍照片 EXIF 顛倒** | 產縮圖時 `exif_transpose` 轉正 |
| 原站 `<title>` 與內文標題不符/帶日期前綴 | 優先取內文 h3 + 去日期前綴 |
| **日期格式不一(YYYYMMDD vs MMDDYYYY 混用)** | 兩種都認、正規化排序 |
| markdownify **丟掉影片/iframe/PDF** | 佔位符保留在原位置,還原成內嵌 |
| 內文殘留舊 `.php` 連結 | 用 urljoin 正確解析 `../` → 新路由 |
| **索引移除但頁面還在的孤兒文章** | Google/Wayback/日期探測 找回 |
| 原站壞圖(引用但 404) | 偵測、記錄、跳過不阻斷 |
| CSS 裡的 `url()` 背景圖漏抓 | 也解析 CSS 內資產 |

> **Firecrawl 用 LLM 抽欄位,但這些是「內容保存的正確性」問題,LLM 不會幫你轉正照片、不會找回孤兒頁、不會保住影片位置。這就是「認真做搬遷」與「通用抽取」的差別。**

## 2.7 技術棧(.NET,對話已列,我確認)

|---|---|---|

**第一版只針對我最懂、手上就有的站型**:語言前綴對稱、伺服器端渲染 HTML、路徑式 URL(香雲寺就是這型)。

# Part 3 — 工程審查補充(開發者視角)

> 以工程師角度審完 Part 1/2 後補上的缺口。按「不先定會返工的程度」分三級:
> **A 級 = 影響架構、0.1 骨架前就要定;B 級 = 應納入 MVP;C 級 = backlog 記下即可。**

## 3.1 【A】字元編碼處理(規劃書原本完全沒提,可能是最大漏洞)

舊 PHP 中文站有很高機率是 **Big5 或 GB2312**,不是 UTF-8,且常見「HTTP header 說一套、meta charset 說另一套、實際位元組又是第三套」。香雲寺剛好是 UTF-8 所以沒踩到,但這是「多語言舊站」工具的必考題:

- 抽取管線第一步就要做**編碼偵測與正規化**(header → meta → BOM → 統計偵測,衝突時以實際解碼成功率為準)。
- **Phase 0 診斷必須報告編碼**,寫進 config(`site.encoding`),讓使用者可覆寫。
- 列入 §2.6 坑清單同級的內建功能;fixture 測試站(見 3.5)要含一頁 Big5 頁面。

## 3.2 【A】Phase 之間的輸出契約(檔案格式先定義)

「可離線重跑」的前提是每個 Phase 的輸出是**明確的檔案格式契約**:

| 契約 | 內容 |
|---|---|
| Phase 1 → 2 | 鏡像目錄結構規範 + `url_inventory` 欄位 schema |
| Phase 2 → 3 | Markdown/frontmatter 欄位規範 + `content_inventory` 欄位 schema + 配對結果格式 |
| 通用 | 各清單 CSV 的欄位名、型別、排序規則,版本化 |

契約定了,才能「只重跑 Phase 2 不重爬」,也才是 NuGet lib 元件化(§2.2)的基礎。**在 0.1 骨架階段就寫成文件**(`docs/contracts.md`)。

## 3.3 【A】語言代碼標準化(i18n-first 工具的門面)

§2.5 的 `lang_prefix: [ch, en]` 裡,`ch` 是香雲寺的 URL 前綴,**不是標準語言代碼**(標準是 BCP-47 的 `zh-Hant`)。輸出 frontmatter 若直接用 `ch`,下游 Astro/Hugo 的 i18n 都對不上。config 改為「URL 前綴 → 標準 locale」對映:

```yaml
url_pattern:
  lang_map: { ch: zh-Hant, en: en }   # URL 前綴 → BCP-47
  default_lang: zh-Hant
```

frontmatter 與清單一律輸出標準 locale;URL 前綴只是來源站的實作細節。

## 3.4 【A】Windows 檔案系統地雷(slug 產生器一開始就要處理)

輸出是幾百個以 URL 命名的檔案,跨平台一致性必須由 slug 產生器保證:

- 260 字元路徑上限(或宣告需要 long path 支援)。
- Windows 保留字檔名(`con`、`aux`、`nul`、`prn`…)。
- 大小寫不敏感檔案系統上的 slug 碰撞偵測。
- Unicode 正規化統一 NFC。

**驗收標準:同一份 config 在 Windows 與 Linux 跑出完全相同的檔案清單。**

## 3.5 【A】測試策略:離線 fixture 站 + golden-file 測試

香雲寺 4.6GB 真實資料進不了 CI,也不能每次測試都打人家網站。做法:

- 建一個**幾十頁的合成雙語小站**放在 test assets,刻意包含每一個 §2.6 的坑:EXIF 顛倒圖、冒號標題、不對稱檔名(活動型)、壞圖 404、iframe/影片、Big5 頁、`%20` 路徑、孤兒頁……
- 配 **golden-file 快照測試**:fixture 站跑完整管線,輸出與 committed 基準逐 byte 比對。
- **每修一個坑就在 fixture 站加一頁** —— fixture 站本身就是功能的活文件。
- 香雲寺真實資料留作本機的 full-scale 迴歸測試,不進 CI。

## 3.6 【B】Verify 階段(把「全站巡檢」做成產品功能)

香雲寺做過「全站巡檢 0 錯誤」,但巡檢本身沒被規劃成功能。加 **`polymigrate verify`**(可視為 Phase 4):

- 檢查:內部連結、圖片/媒體引用、雙語配對完整性、frontmatter 必填欄位。
- 產出巡檢報告。「0 錯誤成績單」是行銷主軸(§2.10),驗證器就該是產品的一部分。

## 3.7 【B】依賴風險(兩項需注意)

| 依賴 | 風險 | 決定 |
|---|---|---|
| **Abot** | 多年低度維護(§2.7 也自承「可能要補」) | MVP 需求只有「序列、延遲、帶 cookie」→ **直接用 `HttpClient` 自寫禮貌爬蟲**,不賭半棄坑套件,也免掉 Windows 包 wget 的麻煩 |
| **ImageSharp** | Six Labors 是**分級授權**,非純 Apache 2.0;商業使用者可能踩授權門檻 | ~~文件明講,或評估 SkiaSharp / Magick.NET 替代~~ **已定案(2026-07-18):風險應驗——ImageSharp 4.x 起「建置」即要求授權金鑰,對開源貢獻者是硬摩擦。已改用 Magick.NET(Apache 2.0),內建 AutoOrient,縮圖尺寸與 Pillow 版實測一致** |

另:**專案本身授權(MIT / Apache-2.0)在 1.0 前定案**,並過一輪依賴授權相容性。

## 3.8 【B】錯誤分級與執行報告(known_issues.md 自動化)

香雲寺的 `known_issues.md` 是手寫的;工具應把它變成自動產物:

- 每次執行輸出**執行報告**:warning / error 分級、壞圖清單、配不起來的頁面、跳過原因。
- `--dry-run` 模式:只報告將發生什麼,不寫檔。
- 明確的 exit code(0 = 乾淨、1 = 有 warning、2 = 有 error),CI 可直接接。

## 3.9 【B】Config schema 驗證與版本

YAML config 是產品主介面:

- 欄位名打錯要立刻報錯並提示「你是不是想打 X」,不可默默忽略。
- 加 `config_version: 1`,格式演進才有退路。
- config 內含 cookie/auth 等秘密 → 文件提醒勿 commit,支援從環境變數注入。
